OpenSearch (Logging)
Pattern ELT: il dato viene estratto in formato grezzo e caricato nel Data Lake prima di essere trasformato. Le trasformazioni sono catalogate in dbt e versionate tramite Project Nessie, permettendo rollback e gestione di ambienti multipli (dev / prod). Kestra orchestra e monitora ogni fase della pipeline. Tutto il codice (pipeline, trasformazioni, configurazioni infrastrutturali) è ospitato e versionato in GitLab, che gestisce anche i deployment via CI/CD.
03 — Componenti della piattaforma
MinIO
Storage
Object storage S3-compatible. Cuore fisico del Data Lake. Cluster da 4 nodi con erasure coding per ridondanza. ~32 TB utilizzabili.
S3 API Erasure Set Bucket Notifications Versioning Data Tiering LDAP / OpenID
Apache Iceberg
Table Format
Table format open source per il Data Lake. Abilita transazioni ACID su object storage, schema evolution e time travel su Parquet/ORC.
ACID Schema Evolution Time Travel Version Rollback Hidden Partitioning
Project Nessie
Table Catalog
Catalogo transazionale open source con versionamento git-like del Data Lake. Gestisce branch e merge su insiemi di tabelle Iceberg.
Git-like branching Multi-table versioning Metadata catalog S3 / MinIO backend
Dremio
Query Engine
Data virtualization layer. Espone interfaccia SQL unificata su fonti eterogenee. Cache in-memory (Reflections). Cluster coordinator + 3 executor.
ANSI SQL Arrow Flight ODBC/JDBC VDS / PDS Row/Col Security Semantic Layer Data Lineage
dbt
Transformation
Repository centralizzato di tutte le trasformazioni SQL/Python. Integrazione nativa con Dremio e Nessie. Abilita testing, documentazione automatica e version control delle pipeline di trasformazione.
SQL + Python Macros Data Tests Auto-docs Nessie branching
Kestra
Orchestrator
Piattaforma di orchestrazione scelta per il progetto (non nel documento originale). Gestisce scheduling, dipendenze, monitoring e alerting di tutte le pipeline ELT della piattaforma.
DAG / Flow Scheduling Event triggers RBAC Alerting Web UI
OpenMetadata
Data Catalog
Catalogo dati unico per tutto il patrimonio informativo regionale. Fonte di verità per metadati, lineage, qualità del dato e governance. Sostituisce DataHub nella configurazione adottata.
Data Discovery Data Lineage Data Quality Ownership / RBAC Webhook notifiche dbt integration LDAP / SSO DCAT-AP
Apache Superset
Visualization
Piattaforma BI open source. Affianca Power BI per la visualizzazione interattiva dei dati. Supporta embedding di dashboard in applicazioni gestionali. Connessione diretta a Dremio via sqlalchemy_dremio.
Dashboard Chart Builder Row-level security Embedding React LDAP / OAuth
Jupyter
Exploration
Ambiente notebook per data exploration, analisi interattiva e prototipazione. Accede a Dremio tramite SQL/Arrow Flight. Stack Python: Pandas, NumPy, GeoPandas, Leafmap, Seaborn.
JupyterHub JupyterLab Python stack Geo analysis Leafmap
OpenSearch
Monitoring
Log aggregator centralizzato per tutti i nodi e componenti. Raccoglie metriche via Metricbeat (agenti push). Fornisce dashboard, alerting e analisi dei pattern di sistema.
Full-text search Metricbeat agents Alerting ML anomaly detection Free on-prem

05 — Scelte adottate rispetto al documento originale
Data Catalog: scelto OpenMetadata (DataHub escluso). Motivazione: architettura più semplice, pull-based ingestion, integrazione nativa con dbt, supporto LDAP/SSO/SAML.

Orchestratore: scelto Kestra (non presente nel documento originale) in sostituzione delle alternative valutate (Airflow, Mage, NiFi). Kestra copre orchestrazione, ELT management, scheduling e monitoraggio in un unico strumento con UI moderna e paradigma dichiarativo YAML.

Source Control & CI/CD: aggiunto GitLab come componente trasversale. Unica istanza per tutti gli ambienti (dev, test, prod). Ospita repository infrastrutturali (Helm charts) e applicativi (progetti dbt). Gestisce pipeline CI/CD e Docker Registry per le immagini dei componenti.

Componente Kafka: non implementato. L'ingestion avviene direttamente tramite Kestra senza message broker intermedio.

Componente Spark: non implementato. Il processing è delegato al motore interno di Dremio (con reflections) e alle trasformazioni dbt.

Notebook: scelto Jupyter (Zeppelin escluso) come ambiente di data exploration interattiva.
